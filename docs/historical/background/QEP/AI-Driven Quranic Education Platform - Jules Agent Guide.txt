# AI-Driven Quranic Education Platform - Jules Agent Guide

## 0. Introduction, Platform Architecture, and Operational Framework

This section provides AI Agent Jules with a comprehensive operational overview of the AI-Driven Quranic Education Platform. Understanding the platform's core concept, vision, strategic objectives, technology stack, architectural structure (including the 15 Core Features and standardized folder layout), and the mapping of original development phases to these features is crucial for your autonomous tasks in automation, monitoring, content management, and system interaction.

### 0.1 Project Concept, Vision, and Strategic Objectives for Agent Context

**Platform Concept & Vision (for Jules):**
The AI-Driven Quranic Education Platform is an ecosystem you help manage and enhance. Its goal is to provide a global, immersive, and ethically sound resource for Quranic learning (recitation, memorization, understanding, living by its teachings). Your operations support this by enabling the integration of AI, ML, AR/VR, and collaborative tools. Key operational parameters for you include upholding Sharia compliance in automated data handling, facilitating personalized learning by managing data flows for AI agents, and ensuring system configurations support universal accessibility.

**Strategic Objectives (Implications for Your Tasks):**
Your automated scripts and monitoring tasks will directly contribute to or be constrained by these objectives:
1.  **Universal Accessibility:** Your scripts may manage multilingual content deployments (e.g., new language packs for i18n), execute accessibility compliance checks (e.g., via linters or specific test suites), or monitor services critical for low-bandwidth performance and PWA functionality.
2.  **AI-Powered Personalized Learning:** You will interact with AI microservices, manage data pipelines that feed personalization algorithms (e.g., learning path recommendations, Tajwīd feedback systems), and potentially monitor the performance and output quality of these AI models.
3.  **Adaptive UI/UX:** You may process user profile update events or emotion data streams (via Kafka topics) that are inputs for UI adaptation logic. Your understanding of these data flows is key.
4.  **Immersive Education Tools:** Your responsibilities might include monitoring AR/VR asset pipelines, validating asset integrity upon deployment, or tracking usage analytics for these experiences.
5.  **Connected Global Communities:** You will assist in automated content moderation workflows, manage community event data (scheduling, reminders), and process group interaction metrics for analysis by other systems (e.g., Swarm Intelligence).
6.  **Sharia & Ethics Compliance:** Your automation scripts handling user data or content must operate within defined ethical boundaries and Sharia guidelines. You might assist in generating logs or flagging data relevant for Sharia compliance audits.
7.  **Cross-Device Compatibility & Privacy/Security:** You may execute automated tests (e.g., UI tests on different viewports) to ensure compatibility or monitor system logs for security alerts related to data privacy (e.g., unauthorized access attempts).

### 0.2 Proposed Technology Stack (Key Interaction Points & Configs for Jules)

Your operational environment and tasks will interface with the following technologies. Understanding key configuration files and interaction patterns is essential:

*   **Monorepo Management (Yarn Workspaces/Nx):** You may need to execute scripts (e.g., `yarn workspace <service> build`) or access configuration from the root `package.json` or specific workspace `package.json` files.
*   **Frontend (React/Vite/Next.js, Tailwind CSS):** For UI testing tasks, you'll interact with the build output or a running instance. Localization files for `react-i18next` (e.g., `/public/locales/`) are key for multilingual content management.
*   **Backend (NestJS - Main Gateway):** This is a primary API interaction point. You will call its documented endpoints (OpenAPI/Swagger specs) for data operations and triggering actions. Monitor its health via `/health` endpoints.
*   **Microservices (Python/FastAPI, Node.js/Express/NestJS):** Located in `/services`, these are critical interaction points. Each should have an OpenAPI/Swagger definition for its API, a `Dockerfile` for its build process, and an `.env.example` for its configuration variables. You will call their APIs and monitor their individual health.
*   **Databases (MySQL, MongoDB, Redis):** Direct interaction should be minimized and controlled (e.g., via specific scripts). You might monitor backup scripts, check data integrity via service APIs, or clear Redis caches. Key configs: connection URIs in environment variables.
*   **AI/ML (LLMs, Speech APIs, Affectiva, Vector DBs):** You will orchestrate calls to these services (often via the `/services/llm-nlp/` or `/services/guidance-assistant/` microservices), manage API keys (from secure environment variables), process their outputs, and monitor their availability/quotas.
*   **Event-Driven Architecture (Apache Kafka):** Critical for your operations. You will consume events from various topics (e.g., `user_events`, `recitation_analyzed_event`) to trigger actions and may produce events to notify other systems or log activities. Key files: Kafka broker addresses in env vars, topic definitions (e.g., `infra/kafka/topic_definitions.json`).
*   **DevOps & Cloud (Docker, GitHub Actions, Firebase, GCP/AWS free tiers):** You will operate within CI/CD pipelines (defined in `.github/workflows/*.yml`), potentially triggering deployment scripts (`scripts/deploy_service.sh <service_name> <env>`) or monitoring cloud resource health via cloud provider APIs or CLI commands.

#### 0.3.2 Standardized Folder/File Structure (Key Paths for Your Operations)

*   **Root Directory Structure:**
    ```
    /
    ├── features/             # Houses code for the 15 Core Features (less direct interaction for you)
    ├── shared/               # Common libraries & components (monitor for changes that might affect services)
    ├── services/             # Your primary interaction targets: standalone microservices
    │   ├── llm-nlp-service/  # Example: Python/FastAPI service for LLM tasks
    │   ├── guidance-assistant-service/ # Example: Node.js service for STT/TTS, emotion
    │   └── ...               # Other microservices (recitation, video, etc.)
    ├── agents/               # Your dedicated directory
    │   ├── jules_scripts/    # Your operational Python or Bash scripts
    │   ├── prompts_library/  # Version-controlled prompts for LLM interactions
    │   ├── config/           # Your specific configuration files
    │   └── worklogs/         # Your operational logs and reports
    ├── infra/                # Infrastructure as Code (Docker, Kafka configs, Terraform)
    │   ├── docker/           # Dockerfiles and docker-compose files
    │   ├── kafka/            # Kafka topic definitions, schemas
    │   └── terraform/        # Terraform scripts for cloud resources
    ├── docs/                 # All project documentation, including this guide
    ├── scripts/              # General utility/automation scripts (e.g., `run_ci_checks.sh`, `deploy_all_services.sh`)
    └── ...
    ```

*   **Microservice Folder Structure (e.g., `/services/<microservice_name>/`):**
    These are key for your monitoring and interaction tasks.
    ```
    /services/<microservice_name>/
    ├── app/                  # Service code (e.g., Python/FastAPI or Node/Express)
    ├── tests/                # Service-specific tests (you might trigger these)
    ├── Dockerfile            # Used by CI/CD and local Docker Compose
    ├── requirements.txt or package.json # Dependencies
    ├── .env.example          # Environment variable template
    └── openapi.json          # API specification for your interaction
    ```

*   **Agent Specific Directory (`/agents/`):** This is your primary workspace.
    *   `/agents/jules_scripts/`: Contains scripts you execute for various tasks (e.g., `monitor_kafka_topics.py`, `generate_usage_report.sh`).
    *   `/agents/prompts_library/`: Stores version-controlled text files or JSON structures defining prompts used for interacting with LLMs (Feature 10).
    *   `/agents/config/`: Your specific configuration files (e.g., `jules_config.yaml` defining alert thresholds, API endpoints you use).
    *   `/agents/worklogs/`: Directory where you should store your operational logs, generated reports, and error logs.

*   **Orig. Phase 2 (User Authentication):** Maps to **F1: User Authentication & Profile Management**. Monitor `user_events` Kafka topic.
*   **Orig. Phase 3 (Billing):** Maps to **F14: Secure Billing, Subscriptions & Donations**. Monitor `payment_events` Kafka topic.
*   **Orig. Phase 4 (Recitation Coach):** Maps to **F2: AI Tajwīd Coach**. Monitor health of `recitation-analysis-service`.

*   **Orig. Phase 8 (Teaching - Educator):** Maps to **F6: Educator Platform & Class Management**. Generate automated reports for teachers.
*   **Orig. Phase 9 (Community):** Maps to **F7: Community Engagement Platform**. Assist content moderation via `/moderate/content` API; monitor `new_community_post_event`.

*   **Orig. Phase 11 (VR/AR):** Maps to **F9: Immersive AR/VR Experiences**. Monitor AR/VR asset integrity and usage.
*   **Orig. Phase 12 (Settings, i18n, PWA):** Maps to **F13: Platform Settings, Globalization & Accessibility**. Manage localization resources; test PWA functionality.
*   **Orig. Phase 13 (Admin, Security):** Maps to **F15: Admin, Analytics & Platform Governance**. Assist admin tasks; monitor security alerts.
*   **Orig. Phase 14 (AI Core Hub):** Maps to **F10: AI Guidance & Fitrah Profiling System**. Key interaction hub for you with LLMs, Affectiva, "Open Computer" agent. Orchestrate via `/services/guidance-assistant-service/`.
*   **Orig. Phase 15 (Swarm, Accessibility, Adaptive UI):**
    *   Swarm Intelligence -> **F11: Swarm Intelligence & Collaborative Learning**. Consume `user_interaction_events` from Kafka; trigger `/services/aco-optimizer-service/`.

    *   Testing (Orig. 16): You execute automated test suites (`scripts/run_all_tests.sh`).

    *   Documentation (Orig. 18): You manage and update content in `/docs` as instructed.
    *   Monitoring (Orig. 19): Central to your role, links to **F15** and involves consuming logs/metrics from all services.

## 1. User Authentication & Profile Management
*   **Agent Interaction Scope:** Jules may be involved in monitoring user registration events (e.g., via Kafka topic `user_created_event`), flagging suspicious account activities based on patterns (e.g., multiple failed logins from `user_login_failed_event`), ensuring data consistency between the primary user database and any auxiliary auth systems (e.g., Firebase for OAuth), and processing profile update events (`profile_updated_event`) to trigger downstream actions like cache invalidation or learning path recalculations.
*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** Kafka topics: `user_events` (includes `user_created`, `user_login_failed`), `profile_change_events`. API access to `/admin/users/` for fetching user details or flagging accounts. User profile data from `/users/profile/{id}` for analysis.
    *   **Outputs:** Alerts to `security_alerts_topic` or admin notification system. Commands to other services (e.g., clear user cache for Feature X). Reports on registration velocity or anomaly detection, stored in `/agents/worklogs/auth_reports/`.
*   **Potential Scripts/Commands/APIs & Configs:**
    *   Scripts: `/agents/jules_scripts/auth/monitor_failed_logins.py`, `/agents/jules_scripts/profile/process_profile_updates_for_cache.py`.
    *   APIs: `GET /admin/users?filter=suspicious`, `POST /admin/users/{id}/flag`.
    *   Config: `/agents/config/auth_monitor_config.yaml` (defining thresholds like `LOGIN_FAILURE_THRESHOLD_PER_HOUR`), `/shared/config/kafka_topics.json`.
*   **Decision Logic:**
    *   IF count(consumed_event from `user_login_failed_event` WHERE userId=X) > LOGIN_FAILURE_THRESHOLD_PER_HOUR THEN produce_event(topic=`security_alerts_topic`, {type: "SUSPICIOUS_LOGIN_ACTIVITY", userId: X}).
    *   ON event(topic=`profile_change_events` WHERE userId=X AND changed_fields CONTAINS 'learning_preferences') DO call_api_to_service('learning-path-service', `/recalculate/{userId}`).
*   **Original Phase Mapping Relevance:** Corresponds to original Phase 2. Your automation scripts for this feature would be part of your tasks related to user lifecycle management and security monitoring. These scripts should be tested as part of the deployment sequence for F1. You might use scripts in `/scripts/test_auth_apis.sh` for this.

## 2. AI Tajwīd Coach
*   **Agent Interaction Scope:** Jules could monitor the health and performance of the AI Tajwīd analysis microservice (expected at `/services/recitation-analysis-service/`), manage queues for audio processing if asynchronous (e.g., via a Kafka topic like `recitation_audio_pending_analysis`), aggregate recitation statistics for reporting to F15, and potentially trigger personalized feedback or encouragement based on performance trends by producing events to `user_notification_topic`.
*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** Recitation submission events (from `recitation_submitted_event` Kafka topic, including audio metadata, user ID), AI analysis results (from `recitation_analyzed_event` Kafka topic, including feedback JSON, scores), AI service health metrics (from `/services/recitation-analysis-service/health` endpoint or Prometheus/Grafana via F15).
    *   **Outputs:** Reports on AI service uptime/latency (to `/agents/worklogs/tajwid_service_health/`), aggregated recitation accuracy reports (for F15), events to `user_notification_topic`.
*   **Potential Scripts/Commands/APIs & Configs:**
    *   Scripts: `/agents/jules_scripts/tajwid/monitor_recitation_service_health.sh`, `/agents/jules_scripts/tajwid/generate_recitation_stats_report.py`.
    *   APIs: `/services/recitation-analysis-service/health`, `/services/recitation-analysis-service/metrics`. Backend APIs like `/admin/recitation/platform-stats`.
    *   Config: `/services/recitation-analysis-service/.env` (for its endpoint, if called directly), `/agents/config/tajwid_monitor_config.yaml` (alert thresholds).
    *   Kafka topics: `recitation_submitted_event`, `recitation_analyzed_event`, `user_notification_topic`.
*   **Decision Logic:**
    *   IF call_api(`/services/recitation-analysis-service/health`).status == "ERROR" FOR 3_CONSECUTIVE_ATTEMPTS THEN create_alert(ops_team, "Tajwid Service Down").
    *   IF consumed_event_from(`recitation_analyzed_event` WHERE userId=X AND score < LOW_SCORE_THRESHOLD) AND count_recent_low_scores(X) > 3 THEN produce_event(topic=`user_notification_topic`, {userId:X, type:"ENCOURAGEMENT_Tajwid"}).
    *   IF new_tajweed_rule_deployed_event received THEN trigger_script(`/agents/jules_scripts/tajwid/request_sample_reprocessing.py`).
*   **Original Phase Mapping Relevance:** Corresponds to original Phase 4. Your monitoring scripts for the `/services/recitation-analysis-service/` are crucial during and post-deployment of this phase.

## 3. Memorization Suite
*   **Agent Interaction Scope:** Jules can automate review reminders by querying for users with items due and producing events to the notification system. It can analyze aggregate memorization patterns (e.g., from `ReviewSessionLog` data) to identify commonly difficult verses for reporting to content teams (F5) or for suggesting adjustments to SRS algorithm parameters (managed via F15).
*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** User memorization item data (verseKey, SRS state like `nextReviewDate` from `/memorization/items/due`), daily list of users with pending reviews.
    *   **Outputs:** Events to `user_notification_topic` for review reminders, reports on "most difficult verses" (e.g., stored in `/agents/worklogs/memorization_reports/`), potential alerts if SRS scheduling tasks fail.
*   **Potential Scripts/Commands/APIs & Configs:**
    *   API calls to: `/memorization/due-reviews/today`, `/analytics/memorization/verse-difficulty-stats`.
    *   Scripts: `/agents/jules_scripts/memorization/send_srs_reminders.py`, `/agents/jules_scripts/memorization/analyze_difficult_verses_report.py`.
    *   Config: `/agents/config/memorization_config.yaml` (reminder schedule).
    *   Kafka topics: `review_session_completed_event`, `user_notification_topic`.
*   **Decision Logic:**
    *   ON schedule (e.g., daily at 7 AM user's local time via F13 settings) FOR EACH user_X in call_api(`/memorization/due-reviews/today`) DO produce_event(topic=`user_notification_topic`, {userId:X, type:"MEMORIZATION_REMINDER"}).
    *   IF call_script(`/agents/jules_scripts/memorization/analyze_difficult_verses_report.py`).output CONTAINS verse_Y with high_difficulty_score THEN notify_content_team(verse_Y, "High Difficulty Reported").
*   **Original Phase Mapping Relevance:** Corresponds to original Phase 6 and parts of original Phase 8 (Memorization Trainer). Your reminder scripts are a key part of this feature's success.

## 4. Gamified Learning & Competitions
*   **Agent Interaction Scope:** Jules can automate the lifecycle of competitions (setup from templates via `/admin/competitions` API, starting/ending based on schedule), update leaderboards in near real-time by consuming score events from Kafka (`recitation_analyzed_event`, `quiz_completed_event`), issue automated rewards or badges via `/users/{id}/grant-badge` API, and monitor for fair play or suspicious activity.
*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** Competition templates/parameters (from `/admin/competitions/templates`), user scores/XP (via Kafka topics like `score_updated_event`, `xp_earned_event`), user reports of unsporting behavior.
    *   **Outputs:** Commands to create/start/end competitions, updated leaderboard data (to Redis cache for F4 UI), events to `user_notification_topic` for achievements/rewards, alerts for fair play violations to admin (F15).
*   **Potential Scripts/Commands/APIs & Configs:**
    *   API calls to: `/admin/competitions` (CRUD), `/leaderboards/recalculate`, `/users/{id}/grant-badge`.
    *   Scripts: `/agents/jules_scripts/gamification/manage_competition_lifecycle.py`, `/agents/jules_scripts/gamification/monitor_fair_play.py`.
    *   Config: `/agents/config/gamification_rules.yaml` (badge criteria, XP rules).
    *   Kafka topics: `competition_lifecycle_event` (e.g., `COMPETITION_START`, `COMPETITION_END`), `user_achievement_unlocked_event`.
*   **Decision Logic:**
    *   IF competition_X.startTime <= NOW AND competition_X.status == 'SCHEDULED' THEN call_api(`/admin/competitions/{id}/start`).
    *   ON event(topic=`score_updated_event` for competition_Y) DO call_api(`/leaderboards/recalculate?competitionId={Y}`).
    *   IF user_Z.xp_total >= BADGE_BRONZE_THRESHOLD AND NOT user_Z.has_badge(BRONZE) THEN call_api(`/users/{Z}/grant-badge?badge=BRONZE`) AND produce_event(topic=`user_notification_topic`, {userId:Z, type:"ACHIEVEMENT_UNLOCKED", badge:"Bronze"}).
*   **Original Phase Mapping Relevance:** Corresponds to original Phase 5 and gamification aspects of original Phase 7. Your automation of competition mechanics is central here.

## 5. Learner-Centric Learning Modules

*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** New/updated content packages (e.g., from a designated Git LFS location or CMS webhook), `user_progress_event` stream, recommendation engine performance metrics (e.g., from `/services/learning-path-service/metrics`), user feedback on content (`content_feedback_event` topic).
    *   **Outputs:** Content validation reports (to `/agents/worklogs/content_validation/`), alerts for broken content, commands to update recommendation model parameters (via API to `/services/learning-path-service/admin/`), reports on content engagement for F15.
*   **Potential Scripts/Commands/APIs & Configs:**
    *   API calls to: `/admin/learning-modules/validate-content`, `/services/learning-path-service/admin/retrain-model`, `/analytics/content/engagement-report`.
    *   Scripts: `/agents/jules_scripts/content_validation/validate_lesson_format.py`, `/agents/jules_scripts/ai/tune_learning_path_model.sh`.
    *   Config: `/agents/config/content_validation_rules.yaml`, `/services/learning-path-service/.env`.
    *   Kafka topics: `learning_content_updated_event`, `user_progress_event`, `recommendation_feedback_event`.
*   **Decision Logic:**
    *   ON new_content_commit_to_cms_branch DO trigger_script(`/agents/jules_scripts/content_validation/validate_new_learning_content.py`).
    *   IF call_api(`/services/learning-path-service/metrics`).accuracy < TARGET_ACCURACY_THRESHOLD THEN trigger_alert("Learning Path AI low accuracy") AND potentially trigger_script(`/agents/jules_scripts/ai/tune_learning_path_model.sh --default-params`).
    *   IF lesson_A.completion_rate_from_kafka_analysis < 5% AND lesson_A.views > 1000 THEN flag_for_content_review(lesson_A) for F6/F15.
*   **Original Phase Mapping Relevance:** Corresponds to original Phase 7 and AI-powered learning path aspects of original Phase 9. Your role in content validation and AI monitoring is key.

## 6. Educator Platform & Class Management

*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** Class rosters, student progress data (from `student_progress_in_class_event` topic or `/educator/classes/{id}/analytics` API), module assignment information, teacher-defined alert criteria (e.g., from `/agents/config/educator_alerts_config.yaml`).
    *   **Outputs:** Automated progress reports (e.g., PDFs stored in `/agents/worklogs/class_reports/` and notification sent), alerts to teachers (via `user_notification_topic`), API calls to F8 to create/schedule video sessions.
*   **Potential Scripts/Commands/APIs & Configs:**
    *   API calls to: `/educator/classes/{id}/progress-summary`, `/educator/classes/{id}/students-at-risk`, `/video/sessions/create` (for scheduling).
    *   Scripts: `/agents/jules_scripts/educator/generate_class_reports.py`, `/agents/jules_scripts/educator/identify_at_risk_students.py`.
    *   Config: `/agents/config/educator_alerts_config.yaml`.
    *   Kafka topics: `student_progress_in_class_event`, `class_schedule_updated_event`, `user_notification_topic`.
*   **Decision Logic:**
    *   ON schedule (e.g., Monday 8 AM as per `/agents/config/cron_schedules.yaml`) FOR EACH active_class DO call_script(`/agents/jules_scripts/educator/generate_class_reports.py` --classId=X) AND send_report_notification_to_teacher(class.teacherId).
    *   IF student_Z.last_active_from_event_stream > 7_days AND student_Z.is_in_active_class_X THEN produce_event(topic=`user_notification_topic`, {userId: class_X.teacherId, type:"STUDENT_INACTIVITY_ALERT", studentId: Z}).
*   **Original Phase Mapping Relevance:** Corresponds to original Phase 8. Automation of reporting and student monitoring are key agent tasks.

## 7. Community Engagement Platform
*   **Agent Interaction Scope:** Jules plays a significant role in assisting human moderators by performing initial AI-powered content moderation (flagging potentially harmful content from `new_community_post_event` topic and routing to `/services/content-moderation-service/`), routing "Ask a Scholar" questions to available scholars (checking `/scholars/availability` API), managing event suggestions (flagging popular ones from `event_suggestion_created_event` topic), and potentially identifying trending topics or unanswered questions for reporting.
*   **Key Data Inputs/Outputs for Agent:**
    *   **Inputs:** `new_community_post_event`, `user_report_content_event`, `event_suggestion_created_event` Kafka topics. Scholar availability status from API.
    *   **Outputs:** Content moderation flags/actions (e.g., API call to `/admin/community/posts/{id}/flag-for-review` or auto-quarantine), notifications to scholars (via `user_notification_topic`), reports on community trends (to `/agents/worklogs/community_reports/`).
*   **Potential Scripts/Commands/APIs & Configs:**
    *   API calls to: `/services/content-moderation-service/moderate`, `/admin/community/posts/pending-approval`, `/scholars/assign-question`.
    *   Scripts: `/agents/jules_scripts/community/process_new_content_for_moderation.py`, `/agents/jules_scripts/community/route_ask_scholar_questions.py`.
    *   Config: `/agents/config/community_moderation_config.yaml` (thresholds for moderation).
    *   Kafka topics: `new_community_post_event`, `user_report_content_event`, `event_suggestion_created_event`, `user_notification_topic`.
*   **Decision Logic:**
    *   ON event(topic=`new_community_post_event`) DO results = call_api(`/services/content-moderation-service/moderate`, post.text); IF results.is_harmful THEN call_api(`/admin/community/posts/{post.id}/quarantine`).
    *   ON event(topic=`ask_scholar_question_event`) DO available_scholar = call_api(`/scholars/get-available`); IF available_scholar THEN call_api(`/scholars/{scholar.id}/assign-question`, question_event.data).

## Shared Area Components
*   **Agent Interaction Scope:** Jules will interact with and rely on many shared components for its own operations. For example, it will use shared logging utilities for its script outputs, leverage shared auth utilities for making secure API calls to platform services, consume global configurations, and potentially use shared notification utilities to send alerts or reports. Automation scripts in `/agents/jules_scripts/` will frequently import or call utilities from `/shared/`.
*   **Key Interactions & Dependencies:**
    *   **UI Component Library (`/shared/ui-components/`):** Not directly used by Jules, but changes here might trigger UI test scripts (e.g., Playwright) located in `/agents/jules_scripts/tests/ui/` that Jules executes.
    *   **Authentication & Authorization Utilities (`/shared/auth/`):** Jules' scripts that call protected APIs will need to use shared token management (e.g., a script to obtain a service account JWT) or client authentication mechanisms defined here.
    *   **API Client Services (`/shared/api-clients/`):** Jules might use standardized API clients/SDKs from this area to communicate with platform services and microservices, ensuring consistent communication patterns and error handling.
    *   **Notification Utilities (`/shared/notifications/`):** Jules will be a heavy user of any shared notification service/client (e.g., an API endpoint `/shared/notifications/send`) to dispatch alerts to admins, operational teams, or users (e.g., memorization reminders).
    *   **Database/ORM Utilities (`/shared/database/`):** If Jules needs to perform read-only data queries for reporting or analysis (and direct DB access is permitted for it, e.g., to a read-replica), it would use shared DB connection configurations or query builder utilities from here.
    *   **Logging Service (`/shared/logging/`):** Jules' scripts and automated processes **must** use the shared logging service/wrapper to ensure its operational logs are centralized and formatted consistently with the rest of the platform. This is key for monitoring Jules' own activities. Log destinations might be defined in `/shared/config/logging_config.json`.
    *   **Constants, Enums & Types (`/shared/constants/`, `/shared/types/`):** Jules' scripts may need to import/use shared constants or enums (e.g., Kafka topic names from `/shared/constants/kafka_topics.js`, event types, user roles) to ensure consistency and avoid hardcoding.
    *   **Build & Configuration Scripts (`/scripts/`, `/shared/config/`):** Jules might trigger or monitor shared CI/CD scripts located in `/scripts/`. It will rely on global configuration files (e.g., `/shared/config/platform_globals.json` or individual files like `/shared/config/service_endpoints.json`) for parameters like API endpoints of various services, alert thresholds etc. Its own specific configurations will be in `/agents/config/`.
*   **Folder Structure Relevance for Jules:**
    *   `/shared/config/`: Critical for Jules to find platform-wide configurations, API endpoints.
    *   `/shared/utils/`: May contain utilities Jules can leverage in its Python/Bash scripts.
    *   `/shared/logging/`: Jules will direct its logs here or use clients that interface with this.
    *   `/agents/config/`: This is Jules' *own* config area, but it might reference master configs in `/shared/config/`.
    *   `/scripts/`: Location of general purpose scripts Jules might execute as part of its tasks.
*   **Decision Logic for Shared Components:**
    *   ALWAYS use `call_shared_script('/shared/logging/log_message.sh', level, message)` for all script output/errors.
    *   IF making API call to `feature_X_service` THEN use `call_shared_script('/shared/api-clients/call_service.sh', service_name, endpoint, payload)`.
    *   LOAD `global_platform_config` from `/shared/config/platform_globals.json` and `/agents/config/jules_specific_config.yaml` at script_startup.
*   **Original Phase Mapping Relevance:** Shared components are developed throughout the lifecycle, but their initial setup and CI/CD integration relate to original **Phase 1 (Project Setup)**. DevOps scripts and configurations in `/scripts/` and `/infra/` relate to original **Phase 17 (Deployment & Optimization)**. Jules relies on these for most of its operations across all features.
