# AI-Driven Quranic Education Platform - Developer Guide

## 0. Introduction, Platform Architecture, and Development Framework

This section provides a comprehensive overview of the AI-Driven Quranic Education Platform, including its core concept, vision, strategic objectives, technology stack, project structure, and the relationship between the original 19-phase development plan and the refined 15 Core Feature model that guides this Developer Guide.

### 0.1 Project Concept, Vision, and Strategic Objectives

**Concept & Vision:**
The AI-Driven Quranic Education Platform aims to be a universally accessible, immersive, and ethically grounded digital solution for Quranic learning. It seeks to empower Muslims of all ages to recite, memorize, understand, and live the Quran. This is achieved by integrating advanced technologies like Artificial Intelligence (AI), Machine Learning (ML), Augmented Reality (AR), Virtual Reality (VR), Video Conferencing (VC), and social learning/gamification features. The platform operates under Sharia compliance within a trust-based framework, emphasizing personalized, emotionally responsive learning facilitated by AI agents and guidance systems.

**Strategic Objectives:**
The platform's development is guided by the following key objectives:
1.  **Universal Accessibility:** Ensuring the platform is usable across diverse devices (mobile, web, AR/VR), network conditions (low-bandwidth, offline via PWA), and by a global audience (multilingual, RTL support).
2.  **AI-Powered Personalized Learning:** Creating dynamic learning pathways that adapt to individual user needs, offering real-time feedback (e.g., Tajwīd correction) and emotion-aware interactions.
3.  **Adaptive UI/UX:** Tailoring the user interface and experience based on user profiles (age, role, skill), cognitive preferences, and emotional states.
4.  **Immersive Education Tools:** Utilizing AR/VR for enhanced visualization and engagement (e.g., AR for Tajwīd, VR for Quranic history).
5.  **Connected Global Communities:** Fostering collaboration and peer learning through competitions, forums, and study groups.
6.  **Sharia & Ethics Compliance:** Adhering to Islamic principles in content and operations, ensuring data privacy (GDPR), and maintaining transparency.
7.  **Cross-Device Compatibility & Privacy/Security:** Ensuring a consistent experience while upholding robust security standards (encryption, RBAC, MFA).

### 0.2 Proposed Technology Stack

The platform leverages a modern, scalable technology stack with an emphasis on free-tier and open-source solutions where viable:

*   **Monorepo Management:** Yarn Workspaces or Nx.
*   **Frontend:** React.js (TypeScript) with Vite or Next.js; Tailwind CSS for styling; Redux Toolkit or React Context for state management; Three.js/A-Frame for AR/VR; D3.js/Recharts for data visualization; `react-i18next` for internationalization.
*   **Backend (Gateway & Core Services):** Node.js (TypeScript) with the NestJS framework.
*   **Microservices:** Python with FastAPI is preferred for AI/ML tasks; Node.js with Express.js or NestJS for other specialized services.
*   **Databases:** MySQL (e.g., AWS RDS Free Tier) for structured data; MongoDB (e.g., Atlas Free Tier) for unstructured data (community content, logs); Redis (e.g., Dockerized) for caching and session management.
*   **AI/ML:**
    *   LLMs: Falcon-Arabic, Llama 3.1 (via Hugging Face Inference API); GPT-4o or similar for advanced guidance.
    *   Speech: Google Cloud STT/TTS, Hugging Face Whisper, Affectiva SDK for emotion detection.
    *   Recitation Analysis: TensorFlow.js, PyTorch.
    *   Vector Databases: Pinecone, Weaviate, or FAISS for semantic search.
*   **Event-Driven Architecture:** Apache Kafka (self-hosted on EC2 free tier or similar).
*   **Authentication:** Custom NestJS implementation with JWT, bcrypt, Speakeasy; Firebase Authentication for OAuth.
*   **DevOps & Cloud:** Docker, Docker Compose, GitHub Actions for CI/CD. Hosting via Firebase Hosting, Vercel/Netlify (frontend), Google Cloud Run, AWS EC2 free tiers (backend/services). Firebase Storage for assets.
*   **Key Libraries/Tools:** Stripe SDK, Helmet.js, Google Perspective API, WebRTC (`simple-peer`), `react-konva`.

The project is organized into a monorepo to facilitate code sharing and streamlined development across different parts of the platform. The development efforts are structured around 15 Core Features, detailed in subsequent sections of this guide.

This guide breaks down the platform into the following 15 core functional areas:

5.  Learner-Centric Learning Modules
6.  Educator Platform & Class Management

#### 0.3.2 Standardized Folder/File Structure

To maintain consistency and predictability across the codebase, the following standardized folder structure is proposed:

*   **Root Directory Structure:**
    ```
    /
    ├── features/             # Contains code for each of the 15 Core Features
    ├── shared/               # Cross-cutting modules, components, and utilities
    ├── services/             # Standalone microservices (AI/ML, specialized tasks)
    ├── agents/               # AI agent (Jules) specific prompts, scripts, logic
    ├── infra/                # Infrastructure as Code (Docker, K8s, Terraform)
    ├── docs/                 # All project documentation
    ├── scripts/              # Utility and automation scripts
    ├── package.json          # Root monorepo package manager
    └── ...                   # Other configuration files (tsconfig, prettierrc, etc.)
    ```

*   **Typical Feature Folder Structure (e.g., `/features/<feature_name>/`):**
    Each core feature module will reside within the `/features` directory and generally follow this internal structure (adaptable for frontend/backend context):
    ```
    /features/<feature_name>/
    ├── api/                  # API route definitions, controllers (backend)
    ├── services/             # Business logic, service classes (backend); data fetching services (frontend)
    ├── ui/                   # React components, pages, hooks, styles (frontend)
    ├── utils/                # Helper functions specific to this feature
    ├── tests/                # Unit, integration, and E2E tests for this feature
    ├── types/                # TypeScript type definitions specific to this feature (or `dto/`)
    ├── index.ts              # Main export file for the feature module (or __init__.py for Python)
    └── README.md             # Feature-specific documentation
    ```
    (Note: For NestJS backend features, this structure would map to its module conventions, e.g., `*.controller.ts`, `*.service.ts`, `*.module.ts`, `dto/`, `entities/`).

*   **Shared Area Folder Structure (`/shared/`):**
    The `/shared/` directory is crucial for DRY principles and houses code accessible by multiple features or services.
    ```
    /shared/
    ├── auth/                 # Auth guards, strategies, utilities (if not part of F1 exclusively)
    ├── database/             # Common database connection utilities, base entities/schemas
    ├── ui-components/        # Globally reusable React components (e.g., buttons, layout)
    ├── utils/                # General utility functions (date formatting, string manipulation)
    ├── config/               # Platform-wide configuration constants or setup
    ├── types/                # Global TypeScript interfaces and types
    ├── hooks/                # Globally reusable React hooks
    └── assets/               # Shared static assets like icons, generic images
    ```

## 1. User Authentication & Profile Management
*   **Purpose & Importance:** This feature is foundational, providing secure access to the platform and managing user identities. It ensures that user data is protected and that experiences can be personalized. For developers, this module underpins most other features, as user context is crucial for personalized content, progress tracking, and role-specific functionalities.
*   **Key Sub-components/Functionalities:**
    *   Secure user registration (email/password, OAuth with Google/Apple).
    *   Login/logout mechanisms.
    *   Multi-Factor Authentication (MFA) setup and verification.
    *   Password reset and account recovery flows.
    *   Session management (JWT-based access and refresh tokens).
    *   Role-Based Access Control (RBAC) implementation (learners, teachers, parents, admins).
    *   User profile creation and management (basic info like name, age range, avatar, learning preferences).
*   **Key Data Entities:** `User` (with fields for credentials, roles, MFA status, profile data), `AuthToken` (for refresh tokens if stored).
*   **Potential APIs/Integration Points:**
    *   Backend (e.g., NestJS): `/auth/register`, `/auth/login`, `/auth/refresh-token`, `/auth/logout`, `/auth/request-password-reset`, `/auth/confirm-password-reset`, `/auth/mfa/enroll`, `/auth/mfa/verify`.
    *   `/users/profile` (CRUD for user profiles).
    *   Protected route guards and decorators for RBAC.
    *   Integration with other services requiring user context (passing user ID, roles).
*   **Folder Structure & Organization:**
    *   Core authentication logic (NestJS services, controllers, DTOs, entities) will reside primarily within the main backend application under a dedicated module, e.g., `/backend/src/auth/` and `/backend/src/users/`.
    *   Frontend components (React/Vite) for login, registration, MFA, and profile editing will be located in `/features/user-authentication/ui/` or `/features/user-profile/ui/`.
    *   Shared authentication utilities like JWT handlers or RBAC decorators, if designed for cross-service use (beyond the main backend), could be in `/shared/auth/`.
*   **Original Phase Mapping:** This core feature directly corresponds to the original **Phase 2 (User Authentication / Advanced User Authentication & RBAC)**. It also encompasses user profile aspects that were implicit in personalization goals across various original phases.

## 2. AI Tajwīd Coach
*   **Purpose & Importance:** This is a core educational feature designed to help users improve their Quranic recitation accuracy according to Tajwīd rules. For developers, this involves handling audio input, interfacing with AI models for analysis, and presenting complex feedback in a user-friendly manner.
*   **Key Sub-components/Functionalities:**
    *   Audio recording and submission interface.
    *   Phonetic analysis of the recitation.
    *   Identification and highlighting of Tajwīd errors (e.g., Madd, Ghunnah, Ikhfa).
    *   Scoring mechanism for recitation attempts.
    *   User-friendly display of feedback.
    *   Storage and retrieval of recitation history and feedback.
*   **Key Data Entities:** `RecitationAttempt` (userId, surahAyahInfo, audioFileUrl, feedbackJson, score, timestamp), `TajweedErrorDetail`.
*   **Potential APIs/Integration Points:**
    *   Backend (NestJS): `/recitation/submit` (uploads audio, calls AI service), `/recitation/history`.
    *   AI Microservice (Python/FastAPI in `/services/recitation-analysis-service/`): `/analyze/tajweed` (input: audio; output: analysis JSON).
*   **Folder Structure & Organization:**
    *   Frontend UI components for audio recording, submission, and feedback visualization will be in `/features/ai-tajwid-coach/ui/`.
    *   The AI analysis engine itself will be a dedicated microservice, located at `/services/recitation-analysis-service/`.
    *   The main backend (NestJS) will have API controllers and services for this feature in `backend/src/features/ai-tajwid-coach/`.
*   **Original Phase Mapping:** This feature directly implements the original **Phase 4 (Recitation / AI Tajwīd Coach - MVP Integration & Enhancement)**.

## 3. Memorization Suite
*   **Purpose & Importance:** This feature supports users in memorizing Quranic verses through scientifically-backed methods like Spaced Repetition Systems (SRS). Developers will need to implement the SRS logic, manage review schedules, and create interfaces for practice and progress tracking.
*   **Key Sub-components/Functionalities:**
    *   Verse selection and goal setting for memorization.
    *   Flashcard-style practice interface (displaying verse, hiding for recall).
    *   SRS algorithm implementation (e.g., SM-2 or FSRS).
    *   Scheduling and management of review sessions.
    *   Tracking memorization progress (verse status: new, learning, mastered, review interval).
    *   Visualizations like heatmaps for memory strength.
    *   AI coaching tips for memorization techniques or difficult verses.
*   **Key Data Entities:** `MemorizationItem` (userId, verseKey, srsData {interval, easeFactor, nextReviewDate, repetitions}, status), `ReviewSessionLog`.
*   **Potential APIs/Integration Points:**
    *   Backend (NestJS): `/memorization/items` (CRUD), `/memorization/review-queue`, `/memorization/log-review`, `/memorization/progress`.
    *   Integration with a notification service (from Shared Area) for review reminders.
    *   AI coaching elements might integrate with F10.
*   **Folder Structure & Organization:**
    *   Frontend components for flashcards, dashboards, and verse selection will be in `/features/memorization-suite/ui/`.
    *   Backend logic for SRS, scheduling, and progress tracking will be in `backend/src/features/memorization-suite/`.
*   **Original Phase Mapping:** This feature directly corresponds to the original **Phase 6 (Memorization / Memorization Suite - SRS & Progress Analytics)** and aspects of original Phase 8 (Memorization Trainer & Scoring).

## 4. Gamified Learning & Competitions
*   **Purpose & Importance:** To increase user engagement, motivation, and retention through game-like mechanics and competitive elements. For developers, this involves creating systems for points, badges, leaderboards, and managing the lifecycle of competitions.
*   **Key Sub-components/Functionalities:**
    *   Points system (XP) for various activities (lessons, recitations, quizzes, community participation).
    *   Badge/Achievement system for milestones and special accomplishments.
    *   Leaderboards (global, regional, by competition type) based on scores or XP.
    *   Recitation and memorization competitions with defined rules, durations, and scoring.
    *   Notifications for competition events and achievements.
*   **Key Data Entities:** `Competition` (title, rules, type, startDate, endDate), `CompetitionEntry` (userId, competitionId, submissionId, score, rank), `LeaderboardCache` (type, scoresJson), `UserXP` (userId, points), `UserAchievement` (userId, badgeId, achievedDate), `AchievementDefinition`.
*   **Potential APIs/Integration Points:**
    *   Backend (NestJS): `/competitions` (CRUD), `/competitions/{id}/join`, `/competitions/{id}/submit-entry`, `/leaderboards/{type}`, `/users/{id}/achievements`, `/xp/grant`.
    *   Real-time updates for leaderboards via WebSockets (potentially through F7 or Shared Area).
    *   Integrates with F2 (AI Tajwīd Coach) and F3 (Memorization Suite) for scoring.
*   **Folder Structure & Organization:**
    *   Logic for managing competitions, XP, and achievements in `backend/src/features/gamification/`.
    *   Frontend components for displaying leaderboards, badges, and competition UIs in `/features/gamification/ui/`.
*   **Original Phase Mapping:** This feature consolidates original **Phase 5 (Competitions / Gamified Competitions & Leaderboards)** and the "Real-Time Feedback & Gamification" aspects of the original Phase 5.

    *   Content authoring/management interface (for admins/educators - likely part of F15 or F6).
    *   Structured learning modules composed of lessons (text, video, interactive exercises) and quizzes.
    *   Quiz engine supporting various question types, automated grading, and feedback.
    *   User progress tracking (completion status per lesson/module, quiz scores).

    *   AI-powered explanations for selected text within lessons (interfacing with F10).

*   **Potential APIs/Integration Points:**
    *   Backend (NestJS): `/learning/modules`, `/modules/{id}/lessons`, `/lessons/{id}`, `/quizzes/{id}/submit`, `/users/{id}/learning-progress`.
    *   AI Microservice (in `/services/learning-path-service/` or `/services/llm-nlp-service/`): `/ai/recommend-learning-path`, `/ai/explain-text`.
    *   Integration with F10 (AI Guidance) for explanations and F11 (Swarm Intelligence) for path recommendations.
*   **Folder Structure & Organization:**
    *   Frontend components for displaying modules, lessons, and quizzes in `/features/learning-modules/ui/`.
    *   Backend logic for content delivery and progress tracking in `backend/src/features/learning-modules/`.
    *   Content authoring tools would be part of F6 (Educator Platform) or F15 (Admin).
*   **Original Phase Mapping:** This corresponds to original **Phase 7 (Learning / Learn-Teach Modules - Learner-Centric Content Delivery)** and aspects of original Phase 9 (AI-Powered Learning Paths).

    *   Educator-specific dashboards and management UIs in `/features/educator-platform/ui/`.

*   **Original Phase Mapping:** Corresponds to original **Phase 8 (Teaching / Learn-Teach Modules - Educator Toolkit & Class Management)**.

## 10. AI Guidance & Fitrah Profiling System
*   **Purpose & Importance:** This is the "Neural Core" of the platform, providing deeply personalized ethical, spiritual, and learning guidance. It involves complex AI model integration (LLMs, emotion AI), conversational AI, a unique Fitrah (innate disposition) profiling system, and voice UI capabilities.
*   **Key Sub-components/Functionalities:**
    *   **Conversational AI (Guidance Assistant):** Using LLMs (Falcon-Arabic, Llama 3.1, potentially GPT-4o) for Quranic Q&A, contextual help, tafsir snippets, and delivering personalized advice.
    *   **Fitrah Profiling:** A system for users to undergo a psychometric assessment (quizzes, interaction analysis) to understand their "Fitrah Spectrum" based on Islamic virtues. This profile is used by the AI to tailor guidance.
    *   **Personalized Recommendations Engine:** Suggests relevant Ayahs, Hadith, Duas, learning modules (from F5), and community interactions (from F7) based on user's Fitrah profile, emotional state, explicit queries, and learning progress.
    *   **Voice UI Control ("Open Computer" Agent):** Enables hands-free interaction with the platform (navigation, settings changes, initiating actions) using natural language commands processed by an AI agent.
    *   **Emotion-Aware Responses:** Guidance and Avatar interactions adapt based on user's emotional state (detected via Affectiva or similar technology, using voice tonality and/or text sentiment).
*   **Key Data Entities:** `UserFitrahProfile` (userId, spectrumScores {aspect: score}), `GuidanceLog` (userId, queryText, voiceCommandTranscript, llmResponse, emotionalContext, timestamp), `AIInteractionLog` (userId, agentType, interactionDetails). (MongoDB is suitable for flexible/unstructured interaction logs).
*   **Potential APIs/Integration Points:**
    *   AI Microservices (Python/FastAPI, located in `/services/`):
        *   `/services/llm-nlp-service/`: Endpoints like `/query`, `/explain`, `/summarize`.
        *   `/services/guidance-assistant-service/`: Endpoints like `/interpret-voice-command` (for Open Computer), `/detect-emotion`.
        *   `/services/fitrah-profiling-service/`: Endpoints like `/assess`, `/get-profile`.
    *   Backend (NestJS, in `backend/src/features/ai-guidance/`): Acts as a gateway, orchestrates calls to AI microservices, manages user context, stores Fitrah profiles and guidance logs. Exposes APIs like `/ai/ask`, `/ai/command`, `/fitrah/assessment`.
    *   Event-Driven Communication (Kafka): For asynchronous tasks like complex Fitrah analysis or logging AI interactions. Topics like `user_guidance_request_topic`, `emotion_detected_topic`.
*   **Folder Structure & Organization:**
    *   Frontend UI for the Guidance Avatar, Fitrah assessment, and voice command interactions in `/features/ai-guidance/ui/`.
    *   The various AI microservices will each have their own directory in `/services/` (e.g., `/services/llm-nlp-service/`, `/services/guidance-assistant-service/`).
    *   Backend orchestration logic in `backend/src/features/ai-guidance/`.
*   **Original Phase Mapping:** This feature is the primary focus of the original **Phase 14 (AI Agents and Guidance Assistant / Neural Core – AI Agents & Swarm / Core Intelligence Hub & Avatar Integration)**. It also incorporates the "Fitrah Profiling & Guidance Bot" aspects from the original Phase 16.

## 11. Swarm Intelligence & Collaborative Learning
*   **Purpose & Importance:** To leverage collective user behavior and data patterns (anonymized and aggregated) to enhance learning for individuals and groups. This involves analyzing large datasets to discover optimal learning pathways, identify common challenges, and provide data-driven insights.
*   **Key Sub-components/Functionalities:**
    *   Collection and aggregation of anonymized user interaction data (e.g., lesson completions, quiz scores, recitation performance, content ratings, common errors in Tajwīd).
    *   Implementation of Ant Colony Optimization (ACO) or similar swarm intelligence algorithms to identify successful learning pathways or frequently accessed content sequences.
    *   Collaborative filtering mechanisms to recommend resources or study partners based on "similar user" patterns.
    *   Providing insights to educators (via F6) on common student difficulties or effective teaching strategies identified from platform-wide data.
    *   Feeding optimized learning path suggestions back into the Learner-Centric Learning Modules (F5) and AI Guidance System (F10).
    *   Real-time data processing and event streaming via Apache Kafka.
*   **Key Data Entities:** `UserInteractionEvent` (userId, eventType, contentId, success_metric, timestamp, metadata - published to Kafka), `AggregatedPattern` (patternId, description, supportingDataPoints), `OptimizedLearningPathSuggestion` (pathId, targetProfile, suggestedModules).
*   **Potential APIs/Integration Points:**
    *   AI Microservice (Python/FastAPI, in `/services/swarm-intelligence-service/` or `/services/aco-optimizer-service/`):
        *   `/swarm/analyze-interactions` (triggered by Kafka events or scheduled).
        *   `/swarm/get-optimized-paths`.
        *   `/swarm/get-collective-insights`.
    *   Kafka Topics: `user_interaction_event_topic`, `learning_progress_event_topic`, `content_feedback_topic`, `optimized_path_ready_topic`, `new_collective_insight_topic`.
    *   Backend (NestJS): Produces relevant user interaction events to Kafka. Consumes insights from the Swarm Intelligence service to update recommendation models or provide data to educator dashboards.
*   **Folder Structure & Organization:**
    *   The core swarm algorithms (ACO) and data processing logic will reside in a dedicated microservice, e.g., `/services/swarm-intelligence-service/`.
    *   Kafka topic definitions and configurations will be in `/infra/kafka/`.
    *   Backend components that publish to or consume from these Kafka topics will be within their respective feature modules (e.g., F5 for learning events).
*   **Original Phase Mapping:** This feature directly maps to the Swarm Intelligence component of the original **Phase 15 (Swarm Intelligence / Accessibility, Offline & i18n / Universal Accessibility (WCAG 2.2) & Adaptive UI/UX Engine)**.

## 12. Adaptive UI/UX Engine
*   **Purpose & Importance:** This feature is crucial for delivering a truly personalized and responsive user experience. It involves dynamically adjusting the UI and UX based on user profiles, preferences, behavior, and even emotional state. Developers will build the frontend logic and backend support to enable these adaptations.
*   **Key Sub-components/Functionalities:**
    *   **Profile-Based Adaptation:** UI changes based on user role (learner, teacher, admin, parent), age group, skill level (beginner, intermediate, advanced).
    *   **Preference-Based Adaptation:** Users can set preferences for learning styles (visual, auditory, kinesthetic), themes (light, dark, Islamic), font sizes, content density, which the UI respects.
    *   **Behavior-Based Adaptation:** The system might simplify navigation or suggest features if a user seems lost, or offer advanced options if a user is highly engaged with a particular tool.
    *   **Emotion-Aware UI (integrates with F10):** Subtle UI cues or content presentation changes based on emotion detected (e.g., offering encouragement if frustration is detected by the AI Guidance system).
*   **Key Data Entities:** `UserProfile` (age, role, skillLevel, preferences {theme, fontSize, learningStyle} - from F1), `UserEmotionalState` (timestamp, detectedEmotion - potentially transient or logged for patterns, from F10).
*   **Potential APIs/Integration Points:**
    *   Frontend: Primarily driven by React Context (UserProfileContext from F1, SettingsContext from F13) and component-level logic to render UI variants. UI components would be designed with adaptive capabilities.
    *   Backend: APIs to save and retrieve detailed user profiles and preferences (`/users/profile`, `/settings`).
    *   AI Microservice (from F10, e.g., `/services/guidance-assistant-service/`): Endpoint to provide emotional state analysis (`/emotion/detect`).
    *   Kafka Topic: `emotion_data_stream` (from F10) for real-time emotional state updates that the frontend (via WebSocket bridge) or a Backend-for-Frontend (BFF) might consume.
*   **Folder Structure & Organization:** Logic for this would be primarily in `/frontend/src/context/` for state management and within individual UI components in `/features/*/ui/` and `/shared/ui-components/` that implement adaptive behaviors. Rules for adaptation might be configurable via F15 (`/features/admin-platform/`). The core engine determining *when* and *how* to adapt based on multiple inputs could be a service in `/features/adaptive-ui-engine/services` or integrated within F10.
*   **Original Phase Mapping:** This maps to the Adaptive UI/UX Engine part of original **Phase 15 (Swarm Intelligence / Accessibility, Offline & i18n / Universal Accessibility (WCAG 2.2) & Adaptive UI/UX Engine)** and is closely linked to F1 (User Profile) and F10 (Emotion Detection).

## 13. Platform Settings, Globalization & Accessibility
*   **Purpose & Importance:** This feature empowers users with control over their experience and ensures the platform is accessible and usable by a global, diverse audience. It covers language support, accessibility features, notification preferences, privacy settings, PWA/offline capabilities, and issuance of certifications.
*   **Key Sub-components/Functionalities:**
    *   **Globalization (i18n & l10n):** Support for multiple languages (e.g., English, Arabic, Urdu, French, Spanish) throughout the UI. Right-to-Left (RTL) language support. Culturally appropriate number/date formatting.
    *   **Accessibility (a11y):** Adherence to WCAG 2.2 AA standards. Keyboard navigation, screen reader compatibility, ARIA attributes, sufficient color contrast, resizable text, focus management.
    *   **User Settings UI:** For managing language, theme, notifications, privacy consents, linked accounts, PWA offline data management.
    *   **PWA & Offline Access:** Service worker implementation (e.g., Workbox) for caching core app shell, essential learning content (lessons, Quran text), and user progress. Offline data storage (IndexedDB) and synchronization mechanisms.
    *   **Certifications & Credentials:** System for generating (e.g., PDF with jsPDF) and allowing users to download/share digital certificates upon completion of modules or learning paths (integrates with F5).
*   **Key Data Entities:** `UserSettings` (userId, language, theme, notificationPrefs {type, enabled}, privacyPrefs {dataSharing}, offlineStorageQuota), `Certificate` (userId, courseId, issueDate, verificationId, pdfUrl).
*   **Potential APIs/Integration Points:**
    *   Backend (NestJS): `/settings` (CRUD for user settings), `/certificates/generate`, `/certificates/user/{userId}`.
    *   Frontend: `react-i18next` for i18n (setup in `/shared/i18n`). Workbox for PWA (setup in `/frontend/src/service-worker.js`). Components for each settings category in `/features/platform-settings/ui/`. Certificate display/download component.
    *   Shared Area: Common i18n utilities, accessibility helper components (e.g., visually hidden text, focus trap) in `/shared/accessibility` and `/shared/i18n`.
*   **Folder Structure & Organization:** Settings UI in `/features/platform-settings/ui/`. Backend in `backend/src/features/platform-settings/`. PWA logic in `/frontend/src/core/pwa/` or `/shared/pwa/`. Certificate templates in `/shared/assets/certificates/`. Localization files in `/frontend/public/locales/`.
*   **Original Phase Mapping:** This feature consolidates original **Phase 12 (Settings / Roles, Permissions & Settings / Globalization, Offline PWA & Core Settings)** and the Universal Accessibility part of original **Phase 15**. Certifications were also a standalone feature mentioned across various planning documents.

## Shared Area Components
*   **Purpose & Importance:** The Shared Area is crucial for developers to maintain a DRY (Don't Repeat Yourself) codebase, ensure consistency across the platform's diverse features, and accelerate development by providing pre-built, reliable common functionalities. It acts as the foundational layer upon which features are built. Its proper organization and use are key to the LEAN-Turbo methodology.
*   **Potential Shared Modules & Contents (referencing `/shared/` subdirectories from Section 0.3.2):**
    *   **`/shared/ui-components/`:** Standardized React components (Buttons, Modals, Forms, Input fields, Layout structures (Header, Footer, Sidebar), Data display elements (Tables, Lists), Loading spinners, Error messages). Built with Tailwind CSS, documented in Storybook. Ensures visual and interaction consistency.
    *   **`/shared/auth/`:** JWT parsing and validation utilities, RBAC helper functions/decorators (backend), `ProtectedRoute` HOC (frontend), OAuth client logic stubs. Used by F1 primarily but also by any service needing to validate tokens or check permissions.
    *   **`/shared/api-clients/`:** Centralized Axios instance or similar HTTP client configuration. Request/response interceptors for token management, error handling, logging. Typed client wrappers or SDKs for interacting with internal microservices.
    *   **`/shared/notifications/`:** Frontend components for toast notifications, in-app alert displays. Backend interfaces for dispatching emails (Nodemailer with free-tier provider), push notifications (Firebase Cloud Messaging - FCM), and in-app notifications (WebSocket-based).
    *   **`/shared/database/`:** Common DB connection utilities (TypeORM/Prisma/Mongoose setup), base repository patterns, shared entity/schema definitions or interfaces (e.g., a base `User` interface).
    *   **`/shared/i18n/`:** Global `react-i18next` setup, language detection logic, core translation files (common terms).
    *   **`/shared/logging/`:** Standardized logging service/wrapper (e.g., Winston for Node.js) for consistent log formatting. Sentry integration for frontend/backend error reporting.
    *   **`/shared/constants/` & `/shared/types/`:** Platform-wide constants, enums (user roles, Kafka topics), global TypeScript interfaces.
    *   **`/shared/validation/`:** Common data validation schemas (Zod, class-validator) or utility functions.
    *   **`/shared/pwa/`:** Core service worker logic (Workbox), IndexedDB wrappers, basic sync utilities.
    *   **`/shared/accessibility/`:** Helper components/hooks for ARIA attributes, focus management.
    *   **`/shared/config/`:** Base configuration loading mechanisms, default platform parameters.
*   **Folder Structure:** The `/shared/` directory will be organized by domain (e.g., `ui-components`, `utils`, `config`, `types`) as outlined in Section 0.3.2. This promotes discoverability and logical grouping of reusable assets.
*   **Phase Mapping Relevance:** Elements of the Shared Area, particularly CI/CD, DevOps tools, and initial configurations, are established in what was original **Phase 1 (Project Setup)** and **Phase 17 (Deployment & Optimization)**. Its continuous development supports all 15 Core Features.
